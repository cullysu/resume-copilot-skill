# Resume Writing Rules

## Bullet Formula

Prefer:

`Strong verb + specific action + scope/context + tool/method + result`

Good patterns:

- Built X using Y to achieve Z.
- Reduced X by Y through Z.
- Led/owned/delivered X across Y scope, resulting in Z.
- Automated X, saving Y time or reducing Z errors.
- Designed X for Y users/devices/workloads, improving Z.

Avoid weak starts:

- Responsible for
- Participated in
- Helped with
- Worked on
- Familiar with

Replace weak starts with what the user actually did: built, led, designed, migrated, automated, analyzed, repaired, negotiated, documented, launched, optimized, validated, configured, diagnosed.

## Metrics

Use confirmed metrics when available. If exact numbers are missing:

- Ask for approximate ranges.
- Use observable outcomes: "cut manual steps from 8 to 2", "supported daily use by a 6-person team", "processed monthly reports", "improved first-pass acceptance".
- Mark uncertain metrics as `needs confirmation`.

Never fabricate numbers. Do not turn "some" into a fake percentage.

## Truthful Enhancement

It is acceptable to:

- Choose the most relevant facts.
- Reorder bullets for the target role.
- Translate informal work into professional terms.
- Convert tasks into impact statements.
- Name tools and concepts the user actually used.

It is not acceptable to:

- Add fake employers, roles, degrees, certifications, publications, or dates.
- Claim leadership, ownership, or scale the user did not have.
- Add tools just because the job description mentions them.
- Hide gaps or issues with misleading dates.

## ATS Guidance

Use standard headings: Summary, Skills, Experience, Projects, Education, Certifications.

Keep important content as selectable text. Avoid tables for final ATS resumes unless the export is known to parse cleanly. Avoid icons as the only label for phone/email/location. Use common file names such as `First_Last_Resume_TargetRole.pdf`.

Match JD vocabulary only when truthful. If the JD says "PostgreSQL" and the user used PostgreSQL, use that exact term. If the user used MySQL, do not rename it.

## First-Time Resume Strategy

For a user with little resume experience:

1. Start with projects and concrete tasks, not abstract self-description.
2. Build a larger source inventory than the final resume needs.
3. Select the strongest 3-5 experiences for the one-page version.
4. Put the clearest role match in the top third.
5. Use a plain, confident tone. Avoid hype words like visionary, world-class, passionate, guru, ninja.

## Review Rubric

Score each section from 1 to 5:

- Relevance: does it match the target role?
- Evidence: are claims backed by facts?
- Specificity: could another person say the same thing?
- Impact: does it show outcomes or meaningful scope?
- Readability: can a recruiter scan it quickly?

Revise the lowest-scoring section first.
